## 1. PROJECT PURPOSE

GoldenGooseTees is a **print-on-demand (POD) adult humor t-shirt store**. Customers:
1. Browse a curated catalog of blank apparel/products
2. Use an in-browser design editor to place text/graphics on the product
3. Approve their design → see a photorealistic Printful-generated mockup
4. Select size/color/quantity
5. Checkout via Stripe
6. Printful handles all printing + fulfillment + shipping

**There is NO custom mockup rendering engine.** All product mockups are generated by the **Printful API v2 mockup-tasks endpoint**. Do not build or reference any custom canvas-based mockup renderer.

## 2. ARCHITECTURE OVERVIEW

```
[Browser]
  └─ Next.js 14 App Router (Vercel)
       ├─ /app/(store)/         ← Customer-facing pages
       ├─ /app/(studio)/        ← Design editor
       ├─ /app/api/             ← API routes (server-side only)
       └─ /components/

[Backend Services]
  ├─ Supabase          ← Auth, PostgreSQL DB, Storage (design files)
  ├─ Stripe            ← Payments, webhooks
  └─ Printful API v2   ← Catalog, mockup generation, order creation

[External]
  └─ Printful          ← Printing + fulfillment + shipping (3PL)
```

### Critical Flow: Design → Mockup → Order

```
1. User designs → Fabric.js canvas (text + PNG graphics)
2. Canvas exported as PNG → uploaded to Supabase Storage
3. Public URL obtained from Supabase Storage
4. POST /api/printful/mockup-task  (server route)
   └─ Calls Printful POST /v2/mockup-tasks with design PNG URL + product/variant IDs
5. Printful returns task_id (async — NOT instant)
6. Printful fires webhook → POST /api/webhooks/printful (mockup_task_finished)
   └─ Server updates DB: designs.mockup_urls[] 
7. Client polls or receives real-time update (Supabase Realtime)
8. User sees mockup → selects options → checkout
9. Stripe payment success webhook → POST /api/webhooks/stripe
   └─ Server calls Printful POST /v2/orders
10. Printful confirms order → order stored in DB with printful_order_id
```

## 4. PRINTFUL API v2 — COMPLETE REFERENCE

**Base URL:** `https://api.printful.com/v2`  
**Auth:** `Authorization: Bearer {PRINTFUL_API_KEY}` (server-side only — NEVER expose in client)  
**Rate limit:** 120 requests/min general; mockup endpoints are lower — always handle 429s with exponential backoff

### 4.1 Key Endpoints Used

#### Get catalog product details + placements
```
GET /v2/catalog-products/{product_id}
GET /v2/catalog-products/{product_id}/mockup-styles
```
Response from mockup-styles includes:
```json
{
  "data": [{
    "placement": "front",
    "display_name": "Front print",
    "technique": "dtg",
    "print_area_width": 12.0,
    "print_area_height": 12.0,
    "print_area_type": "simple",
    "dpi": 150
  }]
}
```

#### Upload a file to Printful (required before mockup OR order)
```
POST /v2/files
Body: {
  "url": "https://your-supabase-storage-url/design.png",
  "type": "default",
  "filename": "design_abc123.png"
}
Response: { "data": { "id": "abc123", "url": "...", "preview_url": "..." } }
```
**IMPORTANT:** Save the returned `id` as `printful_file_id` in the designs table. Use this ID for both mockup tasks AND order creation. Do not re-upload the same file.

#### Create a mockup task (ASYNC)
```
POST /v2/mockup-tasks
Body: {
  "product_id": 71,
  "variant_ids": [4011, 4012, 4013],
  "files": [{
    "placement": "front",
    "file_id": "abc123"
  }],
  "format": "png"
}
Response: {
  "data": {
    "id": 752213059,
    "status": "pending",
    "catalog_variant_mockups": [],
    "failure_reasons": []
  }
}
```
**CRITICAL: Mockups are NOT returned immediately.** Store the task `id` in `designs.mockup_task_id`. Wait for webhook OR poll.

#### Poll mockup task status
```
GET /v2/mockup-tasks?id={task_id}
Response when complete:
{
  "data": {
    "id": 752213059,
    "status": "completed",
    "catalog_variant_mockups": [{
      "catalog_variant_id": 4011,
      "mockups": [{
        "placement": "front",
        "mockup_url": "https://files.cdn.printful.com/mockup/...png"
      }]
    }]
  }
}
```

#### Create an order (called AFTER Stripe payment confirmed)
```
POST /v2/orders
Body: {
  "recipient": {
    "name": "Customer Name",
    "address1": "123 Main St",
    "city": "Johnson City",
    "state_code": "TN",
    "country_code": "US",
    "zip": "37601",
    "email": "customer@email.com"
  },
  "items": [{
    "catalog_variant_id": 4011,
    "quantity": 1,
    "files": [{
      "placement": "front",
      "file_id": "abc123"
    }]
  }]
}
```

### 4.2 Webhook Events to Handle
| Event | Handler Route | Action |
|-------|--------------|--------|
| `mockup_task_finished` | POST /api/webhooks/printful | Update designs.mockup_urls, set status='approved' |
| `package_shipped` | POST /api/webhooks/printful | Update orders.status='shipped', email customer |
| `order_failed` | POST /api/webhooks/printful | Update orders.status, notify admin |

**Webhook verification:** Printful signs webhooks. Always verify the signature before processing.

### 4.3 Common Mistakes — DO NOT DO THESE
- ❌ Never call Printful API from client-side code (exposes API key)
- ❌ Never assume mockup is ready synchronously — always use webhook/polling
- ❌ Never create a Printful order before Stripe payment is confirmed
- ❌ Never re-upload the same design file — reuse `printful_file_id`
- ❌ Never hardcode variant IDs — always load from `products_catalog` table
- ❌ Never skip 429 handling — implement exponential backoff

## 6. DESIGN EDITOR (Fabric.js Canvas)

### Canvas Specifications
- Canvas size: 800×800px (rendered at 72dpi display)
- Export resolution: minimum 150 DPI at print area size (Printful requirement)
- For a 12"×12" print area at 150dpi → export at 1800×1800px minimum
- Background: transparent PNG

### Canvas State Management
- All canvas state stored as Fabric.js JSON (`canvas.toJSON()`)
- Saved to `designs.canvas_json` on every meaningful change (debounced 2s)
- Design PNG exported via `canvas.toDataURL('image/png', 1.0)` then uploaded to Supabase Storage

### Design File Naming Convention
```
supabase/storage/designs/{user_id || 'anon'}/{design_id}/design_final.png
supabase/storage/designs/{user_id || 'anon'}/{design_id}/design_draft_{timestamp}.png
```

### Anonymous → Authenticated Flow
- Anonymous users get a `session_id` stored in localStorage
- Designs are saved with `session_id` and `user_id = null`
- On auth, server-side: `UPDATE designs SET user_id = $1 WHERE session_id = $2 AND user_id IS NULL`
- This must happen in the auth callback (Supabase `auth.onAuthStateChange`)

## 7. STRIPE INTEGRATION

### Payment Flow
1. Customer clicks "Order" → client calls `POST /api/checkout/create-session`
2. Server creates Stripe Checkout Session:
   ```typescript
   const session = await stripe.checkout.sessions.create({
     payment_method_types: ['card'],
     mode: 'payment',
     line_items: [...],
     metadata: {
       design_id: design.id,
       user_id: user?.id || null,
       session_id: sessionId
     },
     success_url: `${origin}/order/success?session_id={CHECKOUT_SESSION_ID}`,
     cancel_url: `${origin}/studio?design_id=${design.id}`,
   });
   ```
3. Create order record in DB with `status: 'pending_payment'`
4. Redirect to Stripe hosted checkout
5. On success: Stripe fires `checkout.session.completed` webhook
6. Webhook handler (`POST /api/webhooks/stripe`):
   - Verify Stripe signature
   - Update order `status = 'paid'`
   - Call Printful `POST /v2/orders`
   - Update order with `printful_order_id` and `status = 'submitted'`

### IMPORTANT: Stripe Webhook Handler Rules
- Always use `stripe.webhooks.constructEvent()` with raw body (not parsed JSON)
- In Next.js, set `export const config = { api: { bodyParser: false } }` on the route
- Handle idempotency: check if order is already 'paid' before re-processing

## 8. SUPABASE CONFIGURATION

### Storage Buckets
| Bucket | Public | Purpose |
|--------|--------|---------|
| `designs` | Yes | User design PNG files (Printful needs public URL) |
| `assets` | Yes | Static product images, brand assets |

### Row Level Security (RLS) Policies
```sql
-- designs table
-- Users can read/write their own designs
CREATE POLICY "user_own_designs" ON designs
  FOR ALL USING (auth.uid() = user_id OR session_id = current_setting('app.session_id', true));

-- orders table  
-- Users can read their own orders only
CREATE POLICY "user_own_orders" ON orders
  FOR SELECT USING (auth.uid() = user_id);
```

## 11. CRITICAL IMPLEMENTATION RULES

1. **Server vs Client boundary:** Any code touching `PRINTFUL_API_KEY`, `STRIPE_SECRET_KEY`, or `SUPABASE_SERVICE_ROLE_KEY` MUST be in `/app/api/` routes or Server Actions. Never in components or hooks.

2. **Mockup is async:** After creating a mockup task, the UI must show a loading state. Use Supabase Realtime to listen to `designs` table changes or poll `/api/printful/mockup-task/[id]` every 3 seconds.

3. **File upload sequence:**
   - User finishes design → export canvas PNG
   - Upload PNG to Supabase Storage → get public URL
   - Call `/api/printful/upload-file` with the Supabase URL → get `printful_file_id`
   - Store `printful_file_id` in DB
   - **Reuse this file_id** for mockup task AND eventual order

4. **Order creation only happens in Stripe webhook handler.** NEVER create a Printful order on the success page or client side. The webhook is the authoritative trigger.

5. **Pricing:** Always calculate final price server-side before creating checkout session. Never trust client-provided prices.

6. **Error handling:** All API routes must return consistent shape:
   ```typescript
   // Success
   { success: true, data: {...} }
   // Error  
   { success: false, error: "Human-readable message", code: "ERROR_CODE" }
   ```
